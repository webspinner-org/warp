# ROADMAP.md — where the Cell is and where it's going

A layman's explanation of what's built, what each surface is for, and
what stands between today and "a Webspinner creates Webspinner without
Claude." Living document; update with each meaningful ship.

---

## The two Looms

The canon names *one* Loom. In practice every Cell has two surfaces
sharing the same code:

- **The admin Loom** (`/admin/*`) — the Wizard's back office. Triage,
  governance, configuration, observability. Nobody but the Wizard
  (or future Wizards on peer Cells) ever sees it.

- **The patron Loom** (everything else, today only `/` as a splash) —
  what a Webspinner who isn't the Wizard sees. They come to a Cell
  to have something *made for them* — a site, a tool, a draft, a
  critique. They don't care about Spinner manifests or vault secrets;
  they want output.

**Today the admin Loom is built. The patron Loom is one splash route
and nothing else.** That is the gap.

## What's running, in plain words

Four services on the Cell host (Kepler, in this Cell's bootstrap):

- **The Grimoire** is the Cell's database (PocketBase). It stores
  everything durable — user accounts, encrypted secrets, the audit
  trail, journal entries, Spinner metadata.
- **The Quiet Loom** is a language model running *on the host*
  (Qwen 14B via mlx). It's the Cell's mind. Nothing it does leaves
  the building.
- **The embeddings sidecar** is a smaller model (MiniLM-L6-v2) that
  turns text into searchable vectors so Spinners can semantically
  recall things.
- **The Loom** (the web app you visit) is the operator surface —
  SvelteKit served by Node, behind a launchd plist.

Four Spinners (specialized workers the Cell hosts):

- **Bootstrap** answers questions about Warp grounded in the canon.
  Audits files for drift. Drafts `DECISIONS.md` entries. Surfaces
  unfinished work.
- **Pablo** reviews UI surfaces against a cited design rulebook
  (WCAG, web.dev typography, NN/g composition, Stephen Few, Webspinner
  brand consistency). He is the design-quality reviewer.
- **Wizard's Journal** keeps an operational diary so a future Claude
  Code session opens already knowing what's going on. The
  `BOOTSTRAP.md` file at the repo root, regenerated by `tools/wj
  bootstrap --write`, is its handoff artifact.
- **Genesis** stands up *another Cell from scratch* on a peer Wizard's
  Mac — probe toolchain, sync repo, build, deploy services. The
  "second Webspinner Wizard joins us in summer" story.

Plus three terminal helpers (`tools/wj`, `tools/audit`, `tools/pablo`),
one deploy helper (`tools/deploy-loom`), and a regression net
(Vitest unit tests + Playwright e2e).

## What each admin screen is for

- **`/admin`** — the landing. "What is this Cell up to right now."
  Version, commit, Spinner count, journal counts, recent commits,
  recent journal entries. Triage and orientation.
- **`/admin/spinners`** — the Skein. List of Spinners installed.
  Future: thousands of Foundation-recognized Spinners discoverable
  for install. Today: the four built-in.
- **`/admin/spinners/<name>`** — each Spinner's detail page. Read the
  docs, invoke the capabilities, see the Silk Pattern (per-Spinner
  history). This is where the *Wizard* tests a Spinner; a patron
  never sees these.
- **`/admin/journal`** — the Wizard's diary, plus the "generate
  session context" button that feeds the next Claude.
- **`/admin/vault`** — encrypted secrets (BYOK keys for Anthropic,
  Resend, Turnstile, etc.).
- **`/admin/audit`** — every Spinner invocation, every privileged
  action. Compliance + diagnostics.
- **`/admin/skein`, `/admin/spools`, `/admin/threads`** — placeholders
  for the Foundation registry, the Spool catalogue, and Warp Thread
  compositions. Stubs today.
- **`/admin/profile`** — the signed-in Wizard.

Significance, plainly: this is the dashboard a Wizard sees when she
runs a Cell. It is not the product. It is the cockpit.

## What gets us to "Webspinner creates Webspinner without Claude"

In rough order:

1. **A patron-facing surface.** The actual storefront — `/` becomes
   a landing where a non-Wizard walks in and is offered something to
   make. Or a separate hostname routes to a patron Loom while
   `kepler.local:3000/admin` stays Wizard-only.

2. **Patron auth (or no-auth).** Today only Wizards register. A
   patron either registers lightly (email + maybe a token), uses
   the Cell anonymously for a free tier, or is invited by a Wizard.

3. **Output-producing Spinners.** Bootstrap / Pablo / Journal /
   Genesis are *operational*. They don't make things for patrons.
   The Foundation's actual offer needs Spinners like
   `weave-website`, `weave-form`, `weave-strategy-doc`, `weave-tool`
   — each one walking a patron through a request and producing an
   artifact. **This is where the architecture's value finally meets
   a non-Wizard.**

4. **BYOK routing on the patron path.** A patron asks for something;
   the Spinner routes through the Wizard's vaulted Anthropic /
   OpenAI / etc. key with full audit; the patron pays via the
   Wizard's relationship. Or the patron BYOKs their own. Both flows
   live in canon §7.

5. **A patron-side "wow" experience.** Splash with the brand. Single
   big text input ("What do you want to make?"). Spinner picks the
   right capability, walks the patron through it, shows progress,
   delivers the artifact. The admin Loom is utilitarian; the patron
   Loom is the showcase.

6. **Distribution.** Either: a patron visits *your* Cell; or
   *every patron becomes a Wizard* with their own Cell (Genesis
   runs from a tarball on their Mac). The Foundation's federation
   story is the second one — every Webspinner has a Cell; Cells
   talk over the Capability Bus.

## What to ship next, concretely

Two candidates depending on which story you want to tell first:

**A. The patron splash + first patron Spinner.**

A non-admin route (say `/weave`) with a single textarea: *"What do
you want to make?"* Behind it, a `weave-website` Spinner that takes
the patron through a short conversation and emits a static-site
bundle. Walks one end-to-end patron flow.

Estimated: 8-12 hours split across the Spinner + the surface +
the artifact emission. **This is the product.**

**B. Patron registration + BYOK routing.**

Stand up the rails — patron accounts in PocketBase, separate from
`users` (or with a `role`), BYOK routing through the vault, audit
chain crossing the Wizard/patron boundary. Then the Spinners that
ride on top become straightforward to add.

Estimated: 6-8 hours. **This is the infrastructure under the product.**

**Recommendation: A first.** Building one real patron flow teaches
more about what the patron Loom needs than speculating about rails.
The first patron Spinner can run on the Wizard's own BYOK key for
now; the multi-tenant story lands when a second patron exists.

The admin Loom stays — every Cell needs it, peer Wizards need it on
summer-Cell day, the Wizard needs it for triage. It just isn't the
product.
