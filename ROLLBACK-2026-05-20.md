# Rollback — 2026-05-20 release

This document captures everything needed to roll the system back to
the state tagged at `warp@v0.7.1` + `webspinner-loom@v0.1.0` on
2026-05-20. The tag is the source of truth; this doc is the runbook.

You should not need this. It exists because v2 (forms-first WYSIWYG)
and v3 (WRAG training) will start landing on top of this state, and
those changes touch new ground (Kepler-local LLM serving, embeddings,
patron data capture). If any of that goes sideways, this is the
quickest path back to a known-good system.

## What "rollback" means

There are three levels, from cheapest to most invasive. Pick the
smallest one that fixes your problem.

### Level 1 — Roll back code only

When: a recent commit broke functionality but data + services + DNS
are fine.

```sh
# On Spindle:
cd ~/warp
git fetch origin
git checkout v0.7.1     # or: git reset --hard v0.7.1 if on a branch
# Sync to Kepler (only the files SvelteKit needs to rebuild):
rsync -av --delete --exclude='.git/' --exclude='node_modules/' \
  --exclude='build/' --exclude='.svelte-kit/' \
  ~/warp/loom/src/ johndavidmarx@192.168.1.132:~/warp/loom/src/
rsync -av --delete --exclude='.git/' \
  ~/warp/tools/ johndavidmarx@192.168.1.132:~/warp/tools/

# On Kepler:
ssh johndavidmarx@192.168.1.132 'bash ~/warp/tools/deploy-loom'
```

Same shape for the loom repo:

```sh
cd ~/webspinner-loom
git checkout v0.1.0
rsync -av --delete --exclude='.git/' --exclude='.venv/' \
  --exclude='__pycache__/' --exclude='*.log' \
  ~/webspinner-loom/ johndavidmarx@192.168.1.132:~/webspinner-loom/
ssh johndavidmarx@192.168.1.132 'launchctl kickstart -k "gui/$(id -u)/com.webspinner.loom"'
```

Cost: minutes. The running services bounce briefly.

### Level 2 — Roll back code + service config

When: a plist change or env var caused a regression and the running
service is unhealthy.

```sh
# Identify the most recent good plist backup:
ssh johndavidmarx@192.168.1.132 'ls -lt ~/.warp-key-rotation/'

# Restore (example for the loom plist):
ssh johndavidmarx@192.168.1.132 'TS=20260520-XXXXXX  # the backup timestamp
  launchctl bootout "gui/$(id -u)/foundation.webspinner.loom"
  cp ~/.warp-key-rotation/$TS/foundation.webspinner.loom.plist \
     ~/Library/LaunchAgents/foundation.webspinner.loom.plist
  launchctl bootstrap "gui/$(id -u)" ~/Library/LaunchAgents/foundation.webspinner.loom.plist'
```

For the hub plist + the demo Loom plist, same pattern. The vault-key
rotation backup directories contain all three.

Cost: a few minutes per service. Sessions that were mid-flight may
need refresh in the browser.

### Level 3 — Roll back the tunnel hostnames

When: `loom.webspinner.ai` or `try.webspinner.ai` ingress is broken
and the new hostname added in this work block is implicated.

```sh
ssh johndavidmarx@192.168.1.132 '
  # Restore the pre-2026-05-20 tunnel config:
  cp ~/.cloudflared/config-prod.yml.bak-20260520-173650 \
     ~/.cloudflared/config-prod.yml
  launchctl kickstart -k "gui/$(id -u)/com.webspinner.cloudflared-prod"
'
```

This removes the `loom.webspinner.ai` ingress and leaves
`try.webspinner.ai` as the only public hostname. DNS for
`loom.webspinner.ai` (the Cloudflare-side CNAME we added via
API) is harmless if left; the tunnel just won't serve that hostname.
To clean it up entirely, delete the DNS record from Cloudflare
dashboard or via API.

Cost: ~10 seconds. Patrons see `loom.webspinner.ai` stop resolving;
`try.webspinner.ai` keeps working.

### Level 4 — Roll back the vault master key

When: vault rotation produced a state where services can't decrypt
secrets and the admin route's recovery isn't working. This is the
most invasive rollback.

```sh
ssh johndavidmarx@192.168.1.132 '
  # Identify the rotation timestamp you want to undo:
  ls -lt ~/.warp-key-rotation/
  TS=20260519-174956   # example — the rotation that needs undoing

  # The backup carries vault_secrets.json (pre-rotation, OLD-key-encrypted)
  # AND the three plists with the OLD master key in them.

  # 1. Restore the plists (this brings the OLD master key back into
  #    the launchd env on next service restart):
  cp ~/.warp-key-rotation/$TS/foundation.webspinner.loom.plist \
     ~/Library/LaunchAgents/
  cp ~/.warp-key-rotation/$TS/foundation.webspinner.loom-demo.plist \
     ~/Library/LaunchAgents/
  cp ~/.warp-key-rotation/$TS/com.webspinner.hub.plist \
     ~/Library/LaunchAgents/

  # 2. Restore the vault rows. The backup has the OLD-key-encrypted
  #    ciphertext + IVs. PATCH each row back to those values:
  PB_EMAIL=$(launchctl print "gui/$(id -u)/foundation.webspinner.loom" \
    | sed -n "s/.*WARP_PB_EMAIL *=> *\(.*\) *$/\1/p" | tr -d " ")
  PB_PASS=$(launchctl print "gui/$(id -u)/foundation.webspinner.loom" \
    | sed -n "s/.*WARP_PB_PASSWORD *=> *\(.*\) *$/\1/p" | tr -d " ")
  TOKEN=$(curl -sS -X POST http://127.0.0.1:8090/api/collections/_superusers/auth-with-password \
    -H "Content-Type: application/json" \
    -d "{\"identity\":\"$PB_EMAIL\",\"password\":\"$PB_PASS\"}" \
    | python3 -c "import sys,json; print(json.load(sys.stdin)[\"token\"])")
  python3 <<PY
import json, urllib.request as r
rows = json.load(open("/Users/johndavidmarx/.warp-key-rotation/'$TS'/vault_secrets.json"))["items"]
for row in rows:
    body = json.dumps({"ciphertext": row["ciphertext"], "iv": row["iv"]}).encode()
    req = r.Request(
        f"http://127.0.0.1:8090/api/collections/vault_secrets/records/{row[\"id\"]}",
        data=body, method="PATCH",
        headers={"Authorization":"'$TOKEN'", "Content-Type":"application/json"})
    print(row["name"], r.urlopen(req).status)
PY

  # 3. Reload all three services:
  for L in foundation.webspinner.loom foundation.webspinner.loom-demo com.webspinner.hub; do
    launchctl bootout "gui/$(id -u)/$L" 2>/dev/null || true
    sleep 1
    launchctl bootstrap "gui/$(id -u)" ~/Library/LaunchAgents/$L.plist
  done
'
```

Patron warp_hub + warp_author cookies signed with the more-recent
key become invalid. Patrons re-sign-in. Admin wp_session (PB token,
not HMAC) survives.

Cost: a few minutes. All HMAC cookies invalidated. Document the
rollback in DECISIONS.md afterward.

## What standalone .html files do during rollback

Already-downloaded standalone files have ZERO dependency on tags or
service states — they reference `app.webspinner.ai` only for the
optional `/version` freshness ping (which fails silently). They keep
working regardless of what gets rolled back, except:

- If you roll back the renderer extraction (#58 in v0.7.1) by going
  to a tag older than v0.7.1, the `/standalone` endpoint will revert
  to v1 hybrid (which references SvelteKit chunks). Existing
  downloaded files keep working.

## Forward-compat: starting v2/v3 work

Before any v2 (forms-first WYSIWYG) or v3 (WRAG training) code
lands on `main`, create a branch off `v0.7.1` so you can iterate
without contaminating the released state:

```sh
cd ~/warp
git checkout v0.7.1
git checkout -b loom-v2-work
```

When v2 ships, tag it `v0.8.0` and merge to main. The branch model
keeps `v0.7.1` reachable as a release pointer + as a known-good
stash should you need to bisect a regression.

For the loom repo, same shape:

```sh
cd ~/webspinner-loom
git checkout v0.1.0
git checkout -b loom-v2-frontend
```

## What's outside this rollback's scope

- **Cloudflare DNS records** beyond the tunnel-ingress change. The
  `loom.webspinner.ai` CNAME we added isn't strictly part of any
  rollback level above — it's a one-time DNS write. To remove it,
  the easiest path is the Cloudflare dashboard. The API token is
  in the operator vault under `webspinner-ai-try`.
- **GitHub repo rename** (`webspinner-try` → `webspinner-loom`).
  GitHub's auto-redirect means the old URL still works; reversing
  the rename via `gh api -X PATCH /repos/.../-f name=webspinner-try`
  is one call but rarely needed.
- **Patron data.** No patron data is touched by any of these
  rollbacks. wp_spinner_sessions, wp_database_applications, and
  wp_app_packages are all preserved.

## When to use which level

| Symptom                                                         | Try first                                               |
| --------------------------------------------------------------- | ------------------------------------------------------- |
| Recent commit broke a route or UI                               | Level 1                                                 |
| `launchctl print` shows a service crashed-and-respawning        | Level 2                                                 |
| Tunnel ingress is returning 502 for `loom.webspinner.ai`        | Level 3                                                 |
| Hub can't decrypt vault on startup; admin route doesn't recover | Level 4                                                 |
| A patron's app is broken but their data is fine                 | Level 1 (and double-check the renderer wasn't reverted) |
| All four — something cascaded                                   | In order: 4, 3, 2, 1 (deepest first)                    |

## After a rollback

Two things you owe the record:

1. **A `DECISIONS.md` entry.** Date, level used, why, what broke,
   what got rolled back. The next session needs to know.
2. **A tag at the rolled-back state** if you intend to ship from it:
   `git tag -a v0.7.2 -m "..." && git push origin v0.7.2`.

The tag matters because it gives `loom-v2-work` (the forward branch)
a sibling to compare against. Without it, "what was the last good
production state" becomes a question whose answer drifts.
